# 任务清单：[FEATURE NAME]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档

**前置条件**：plan.md（必需）、spec.md（用户故事必需）、research.md、data-model.md、contracts/

**测试说明**：下面的示例包含测试任务。测试是可选的，只有在功能规格明确要求时才需要加入。

**组织方式**：任务按用户故事分组，方便每个故事独立实现、独立测试。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可并行执行（不同文件、无依赖）
- **[Story]**：任务属于哪个用户故事（例如 US1、US2、US3）
- 描述里要包含准确的文件路径

## 路径约定

- **单项目**：仓库根目录下的 `src/`、`tests/`
- **Web 应用**：`backend/src/`、`frontend/src/`
- **移动端**：`api/src/`、`ios/src/` 或 `android/src/`
- 下方示例默认按单项目结构写，实际使用时请根据 `plan.md` 调整

## 依赖与执行顺序

### 阶段依赖

- **准备阶段（Phase 1）**：无依赖，可立即开始
- **基础阶段（Phase 2）**：依赖准备阶段完成，并且会阻塞所有用户故事
- **用户故事阶段（Phase 3+）**：都依赖基础阶段完成
  - 如果团队资源允许，可以并行推进
  - 否则按优先级顺序执行（P1 -> P2 -> P3）
- **收尾阶段（最终阶段）**：依赖所有目标用户故事完成

### 用户故事依赖

- **用户故事 1（P1）**：基础阶段完成后即可开始，不依赖其他故事
- **用户故事 2（P2）**：基础阶段完成后即可开始，可能会与 US1 集成，但必须保持可独立测试
- **用户故事 3（P3）**：基础阶段完成后即可开始，可能会与 US1/US2 集成，但必须保持可独立测试

### 每个用户故事内部顺序

- 如果包含测试，必须先写测试并确认失败
- 先 model，再 service
- 先 service，再 endpoint
- 先核心实现，再集成
- 当前故事完成后，再推进下一个优先级

### 可并行机会

- 所有标记为 [P] 的准备阶段任务都可以并行
- 基础阶段里标记为 [P] 的任务也可以并行
- 基础阶段完成后，不同用户故事可以并行推进
- 一个用户故事中的 [P] 测试任务可以并行
- 一个用户故事中的 [P] model 任务可以并行
- 不同成员可以分别负责不同用户故事

## 实施策略

### 先做 MVP（只做用户故事 1）

1. 完成 Phase 1：准备阶段
2. 完成 Phase 2：基础阶段（关键，阻塞所有故事）
3. 完成 Phase 3：用户故事 1
4. **停止并验证**：单独测试用户故事 1
5. 如果已经满足目标，可以直接演示或发布

### 增量交付

1. 完成准备阶段 + 基础阶段，形成基础能力
2. 加入用户故事 1 -> 单独测试 -> 演示/发布（MVP）
3. 加入用户故事 2 -> 单独测试 -> 演示/发布
4. 加入用户故事 3 -> 单独测试 -> 演示/发布
5. 每个故事都应在不破坏前面成果的前提下持续增加价值

### 并行团队策略

多人协作时：

1. 团队先一起完成准备阶段 + 基础阶段
2. 基础阶段完成后：
   - 开发者 A：负责用户故事 1
   - 开发者 B：负责用户故事 2
   - 开发者 C：负责用户故事 3
3. 每个故事独立完成并逐步集成

## 备注

- `[P]` 任务表示不同文件、无依赖，可以并行
- `[Story]` 标签让任务可以追溯到具体用户故事
- 每个用户故事都应能独立完成、独立验证
- 实现前要先确认测试是失败的
- 建议每个任务或每组逻辑任务完成后单独提交
- 每个检查点都可以停下来单独验证当前故事
- 避免：过于模糊的任务、多个任务同时改同一文件、破坏独立性的跨故事强依赖
